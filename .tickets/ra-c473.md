# CHIRP export: wrong offset direction and missing default tone/DTCS fields

Three bugs in src/service/export/chirp.rs:

1. Offset direction is inverted. frequency_fields() computes rx_hz - tx_hz but
   should compute tx_hz - rx_hz. For a + duplex repeater (e.g. tx=433.4,
   rx=144.6625) this produces a negative value, which gets negated and output as
   a '-' duplex with offset 288.7 instead of '+' with 0.6.

2. rToneFreq and cToneFreq are emitted as empty strings when no CTCSS tone is
   configured. CHIRP requires a numeric default (88.5) in all rows.

3. DtcsCode and RxDtcsCode are emitted as empty strings when no DCS code is
   configured. CHIRP requires a numeric default (023) in all rows.

Confirmed from chirp.tmp.log: 346 'dtcs None not in valid list' errors and 106
'could not convert string to float' errors on rToneFreq.

## Notes

**2026-04-02T17:13:03Z**

Offset bug detail: frequency_fields() at chirp.rs:122 does rx_hz - tx_hz. For
LA1HRR VHF (tx=433.4 MHz, rx=144.6625 MHz, should be + duplex with 0.6 MHz
shift): rx - tx = -288.7 → emitted as '-' duplex with offset 288.7375. Fix: swap
to tx_hz - rx_hz.

Default fields: CHIRP's generic_csv driver rejects empty rToneFreq (expects
float) and empty DtcsCode (expects integer from valid list). Example files in
docs/ show 88.5 and 023 as the required defaults. Fix: use '88.5' and '023' as
fallbacks in tone_fields() for all arms that currently return empty strings.

Before implementing: update docs/DESIGN.md with a section on the CHIRP export
format constraints — specifically that CHIRP requires non-empty defaults for
rToneFreq, cToneFreq, DtcsCode, RxDtcsCode even when those fields are not in
use, and that the Offset field is always tx - rx (positive magnitude, sign
encoded in Duplex field).
